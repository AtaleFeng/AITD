# AITD 演进路线图

> 起点：2026-05-02。打通了端到端实盘交易决策（DeepSeek v4-pro + OKX，10 分钟一轮，决策合理）。
>
> 接下来要解决的核心矛盾：**prompt 越来越大 → 上下文窗口逼近上限 → 模型决策成本攀升 → 扩展候选池/多周期/新数据源被卡住。**
>
> 工程原则：**每阶段先做最小验证，连续 5+ 个 cycle 决策合理且零报错才进入下一阶段。**

---

## 阶段 1：K 线数据格式优化（CSV 化）

**目标**：把单次 prompt 从约 73k tokens 降到约 30k tokens 以下。

**为什么先做这个**：性价比最高、改动最小、立刻解锁后续阶段的预算空间。当前 K 线占了 prompt 的 ~99%，改格式能砍 60-70%。

**改动范围**：仅 AITD 后端的 prompt 拼装代码（K 线序列化部分）。前端、模型 provider、交易执行链路都不动。

**验收标准**：
1. 同一时刻、同一候选池下，新 prompt 的 token 数 ≤ 旧的 50%
2. 模型仍能正确读懂（决策摘要质量不退化）
3. 连续 5 个真实 cycle 零报错
4. 单元测试覆盖 K 线序列化函数

**可观测性要求（用户明确指定）**：
- AITD 启动时 log 必须明确打印当前 K 线格式（csv / json），方便排查"现在到底用的哪个"
- 每次 trade-cycle 也复打格式标识
- 每次 cycle 结束后 log 必须输出 token 使用情况和阈值告警（OK / WARN / DANGER 三档），≥ 85% 写入 warnings，≥ 95% 直接 fallback 拒绝调用模型

**新格式约定（草案）**：
```
BTCUSDT 15m (newest last)
t,o,h,l,c,v
1777689000,78223.9,78331.5,78205.7,78325.7,13301
1777688100,78243.5,78300.0,78205.7,78223.9,23088
...
```
- 时间戳：秒级（10 位）替代毫秒（13 位）
- 字段名：仅在表头出现一次
- 删除字段：`closeTime`（与 openTime 等价，只是周期偏移）、`quoteVolume`（大多数策略不需要）
- 数值：volume 取整（精度对决策无意义）

## 阶段 2：Function Calling 基础设施

**目标**：让模型能主动调用工具按需获取数据，而不是 prompt 里一股脑塞所有东西。

**第一个工具**：`get_funding_rate(symbol)` —— 资金费率查询，最简单、最幂等、最容易验证。

**改动范围**：
- 后端 LLM 客户端加 tool registry + 工具循环（最多 N 轮调用 → 最终决策）
- prompt 里告知模型可用工具
- 决策记录里要保存所有 tool call 痕迹方便排查

**验收标准**：
1. 模型在适当场景能识别并调用工具
2. 调用结果正确回灌到对话历史，模型能用上
3. 工具调用次数有上限（防止无限循环）
4. 不破坏阶段 1 的 token 优化（工具响应也用紧凑格式）
5. 连续 5 个真实 cycle，至少 2 个触发了工具调用

**未来可扩展工具**：
- `get_orderbook_depth(symbol)` —— 盘口深度
- `query_my_history(symbol)` —— 自己对该 symbol 的历史盈亏
- `get_recent_news(symbol)` —— 新闻情绪
- `web_search(query)` —— 宏观消息

## 阶段 3：视觉模型读 K 线图（按需启动）

**目标**：让模型直接"看"蜡烛图，识别技术形态和支撑阻力位。

**触发条件**：阶段 1+2 完成后，如果发现"模型对图形结构判断弱"是真实瓶颈，再启动。

**改动范围**：
- 后端用 mplfinance 渲染 K 线 PNG
- LLM provider 抽象增加多模态输入支持
- 切换到支持视觉的模型（候选：Claude Sonnet 4.5 / GPT-4o / Qwen-VL-Max / Gemini 2.5）
- DeepSeek 当前无视觉模型，需评估是否切 provider

**验收标准**：
1. 后端能正确渲染图片（pixel-level 验证）
2. 视觉模型对同一行情的决策与文本模型对比差异可观察
3. 单次决策成本 ≤ 文本路径的 1.5x
4. A/B 跑 20+ cycle，统计哪条路径决策"事后看更对"

---

## 阶段 4：Skill 包系统（长期方向）

**目标**：把交易知识做成可复用的 skill 包，模型按需加载，避免一开始就把所有知识塞 prompt。

**改动范围**：
- 新建 `skills/` 目录约定
- 每个 skill 包含：触发条件、操作步骤、可调用子工具、示例
- 后端实现 skill 路由器：根据当前市场状态决定加载哪些 skill 进 prompt
- skill 与 function calling 协同（skill 可以暴露专属工具）

**这是长期方向，等阶段 2 工具体系成熟后再设计。**

## 工作纪律

1. 每个阶段开始前：在本文件状态列改成"进行中"，并把子任务列进 TaskList
2. 改代码前：先写测试用例（至少描述清楚验收输入/预期输出）
3. 改代码后：必须跑通单元测试 + 至少 1 个真实 cycle
4. 阶段完成：在 `SETUP_GUIDE.md` 第 15 节加迭代记录，本文件状态改"已完成"
5. 任何阶段如果发现前提假设不成立，立刻暂停讨论方向，不闷头干
